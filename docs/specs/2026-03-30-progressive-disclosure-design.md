# Progressive Disclosure for Memory Context Injection

**Date:** 2026-03-30
**Status:** Approved

## Problem

The `mem-context-hook` dumps all facts (37+), all soul entries (25+), and today's logs on every message. Most are irrelevant to any given prompt, wasting context window tokens and adding noise.

## Solution: Tiered Injection

Split the hook output into two tiers:

### Tier 1 - Always Injected (full content)

- All `soul` entries (behavioral rules that should guide every response)
- Facts in categories: `user_info`, `preferences` (identity + workflow config)
- Today's log entries (session-specific, small)

### Tier 2 - Index Only (one-liner per fact)

- All other fact categories: `notes`, `projects`, `people`, `decisions`, `work`, `principle-violations`, `workflow`
- Format: `category/subject: <description>`
- Agent uses `mem query` to fetch full content when needed

### Example Output

```
=== MEMORY CONTEXT [Mon 30 Mar 2026, 14:30] ===

SOUL:
communication_style: prefers direct, concise responses...
commit_message_format: always use conventional commits...
...

USER:
name: Octavian
employer: Does NOT work at Anthropic. Currently Full-Stack Developer at ThirdEarAI.
portfolio-title: Prefers 'Product Engineer' as portfolio title.

PREFERENCES:
beans-cli: Always use the beans CLI...
documentation-discipline: All exports must have TSDoc/JSDoc...

TODAY'S LOG:
(empty)

INDEX (use `mem query "SELECT content FROM facts WHERE category='x' AND subject='y'"` to fetch full content):

projects/raft-cli: TUI for GitHub PR management, Bun + OpenTUI
...
```

## Schema Change

Add `description TEXT` column to the `facts` table. This is a short one-liner (under 80 chars) provided at save time, analogous to skill frontmatter descriptions. Used by the hook for the index tier.

```sql
ALTER TABLE facts ADD COLUMN description TEXT;
```

Fallback: if `description` is NULL, the hook truncates `content` to the first 80 characters for the index line.

## Changes Required

### 1. `mem-init` (bin/mem-init)

Add `description TEXT` column to the facts CREATE TABLE statement. Add an ALTER TABLE migration for existing databases.

### 2. `mem fact` (bin/mem)

Accept optional `--desc "..."` parameter. When provided, store it in the `description` column alongside the fact content.

```bash
# New usage
mem fact notes mobile-chat-jump "Full debugging context..." --desc "Mobile web chat jump bug, PR #697, stuck"

# Without --desc (description stays NULL, hook truncates content)
mem fact notes quick-note "Something short"
```

### 3. `mem-context-hook` (bin/mem-context-hook)

Replace the single "dump everything" query with tiered output:

- **Tier 1 categories** (`user_info`, `preferences`): `SELECT category, subject, content FROM facts WHERE category IN ('user_info', 'preferences')`
- **Tier 2 categories** (everything else): `SELECT category, subject, COALESCE(description, SUBSTR(content, 1, 80)) FROM facts WHERE category NOT IN ('user_info', 'preferences')`
- **Soul**: unchanged, always full content
- **Today's log**: unchanged, always full content

The tier 1 category list should be defined as a variable at the top of the script for easy modification.

### 4. Skill Docs

**cookbook/saving.md**: Add guidance on writing good descriptions when saving facts. Descriptions should be under 80 chars and capture the essential "what is this about" for index scanning.

**cookbook/searching.md**: Add the `mem query` pattern for fetching full content from the index:
```bash
mem query "SELECT content FROM facts WHERE category='notes' AND subject='mobile-chat-jump'"
```

**SKILL.md**: Update the CLI quick reference to show `--desc` parameter.

### 5. `mem dump` / `mem import` (bin/mem)

Include the `description` field in export and import. The dump JSON should include `description` alongside `category`, `subject`, `content`.

### 6. Backfill Existing Facts

Generate descriptions for all 37 existing facts. This is a one-time operation done by the agent during implementation, using direct SQL updates:

```bash
mem query "UPDATE facts SET description='...' WHERE category='...' AND subject='...'"
```

### 7. `mem-extract` (bin/mem-extract)

No changes needed. Auto-extracted facts from regex patterns are simple enough that truncation fallback is fine, or descriptions can be hardcoded per pattern.

## Design Decisions

- **No new CLI commands.** Agents already know `mem query` for SQL. Adding `mem get` would be unnecessary abstraction.
- **Description provided at save time**, not auto-generated by LLM. The caller knows best what a fact is about. This avoids API latency and keeps `mem fact` fast.
- **Tier 1 categories are configurable** via a variable in the hook script, not hardcoded throughout.
- **Fallback to truncation** means the system works without descriptions. No hard dependency on backfill being complete.
- **Soul entries are never indexed** because they're behavioral rules that should always be active. Even with many soul entries, they're short and high-value.
